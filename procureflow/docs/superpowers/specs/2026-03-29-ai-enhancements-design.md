# AI Enhancements Design — Procurement Intelligence, AI Agent, Inventory Forecasting

**Date:** 2026-03-29
**Status:** Draft
**Scope:** 3 AI-driven features to enhance ProcureFlow's automation capabilities

---

## Overview

Three features that make ProcureFlow more AI-driven:

1. **Procurement Intelligence** — Proactive insight cards generated by n8n cron + Claude, shown in the dashboard
2. **AI Agent with Actions** — Upgrade chatbot from read-only to action-capable with confirmation dialogs + RBAC
3. **Inventory Forecasting** — Hybrid demand prediction (WMA default + Claude on-demand) with auto-reorder suggestions

## Architecture: Feature-Specific Services + Shared Toolkit

Each feature gets its own service. A lightweight shared toolkit provides the Claude client, tool definitions, and prompt fragments.

### Shared Toolkit (`lib/ai/`)

**claude-client.ts** (~80 lines):
- Anthropic SDK singleton instance (reusable across all services)
- Retry with exponential backoff (max 3 attempts)
- Rate limiting: 50 req/min (configurable via env)
- Token budget per call (max_tokens)
- Structured logging: model, tokens_used, latency_ms

**tool-registry.ts** (~200 lines):
- READ tools (available to all services), migrated 1:1 from chat.service.ts:
  - `search_requests` (was `search_requests`) — search/filter purchase requests
  - `get_request_detail` (was `get_request_detail`) — fetch full request with items, approvals, timeline
  - `search_vendors` (was `search_vendors`) — search/filter vendors
  - `get_budget_overview` (was `get_budget_overview`) — budget status and usage
  - `get_invoice_stats` (was `get_invoice_stats`) — invoice statistics and totals
  - `search_invoices` (was `search_invoices`) — search/filter invoices
  - `get_inventory_stats` (was `get_inventory_stats`) — material stock levels
  - `get_tender_stats` (was `get_tender_stats`) — tender statistics
- WRITE tools (agent.service.ts only, filtered by RBAC):
  - `create_request` — create new purchase request (allowed fields: title, description, vendor_id, items, priority, needed_by, category, department, cost_center, budget_code)
  - `update_request` — update own request (allowed fields: title, description, priority, needed_by, vendor_id, category — excludes status transitions)
  - `submit_for_approval` — submit request for approval workflow
  - `approve_request` — approve a pending request in queue
  - `create_vendor` — create new vendor record
  - `bulk_update` — update multiple requests (ADMIN only, max 50 per batch, allowed fields: priority, category, department only)
- Each tool: `name`, `description`, `input_schema`, `handler`, `permission_level`

**prompts.ts** (~50 lines):
- Shared system prompt fragments (company context, Italian language, safety guardrails)
- Per-feature system prompts compose from these fragments

### Migration from chat.service.ts

1. All 8 existing read-only tools migrate 1:1 to `tool-registry.ts` (no tools removed, no renames)
2. `agent.service.ts` imports all READ tools from registry + adds 6 WRITE tools
3. `POST /api/chat` → points to `agent.service.ts` (drop-in replacement)
4. `chat.service.ts` → deleted (no other consumer)
5. `suggest.service.ts`, `email-ai-classifier.service.ts`, `invoice-ai-parser.service.ts` → unchanged
6. Module guard retained: `requireModule('/api/chat')` stays in route.ts
7. Insight and forecast endpoints do NOT need module guards (they are core AI infrastructure)

## Feature 1: Procurement Intelligence

### Flow

```
n8n Cron (every 6h) → POST /api/ai/insights → insight.service.ts
  → 5 DB queries (spend, delivery, approvals, budgets, recurring items)
  → Claude structured output → score + deduplicate
  → Save to DB: AiInsight → Dashboard widget reads via GET
```

### Insight Categories

| Type | What It Detects | Example |
|------|----------------|---------|
| **SPEND_ANOMALY** | Spend deviation per vendor vs 3-month average | "Spesa verso Fornitore X +45% rispetto al trimestre scorso" |
| **VENDOR_RISK** | Late deliveries, single-vendor dependency | "3 ordini da TechParts in ritardo negli ultimi 30gg" |
| **SAVINGS** | Duplicate purchases cross-vendor, volume consolidation | "Consolidare ordini carta da 3 fornitori → ~€1.200/anno" |
| **BOTTLENECK** | Slow approvals by department, stuck requests | "Approvazioni IT: 5.2gg media vs 1.8gg company-wide" |
| **BUDGET_ALERT** | Projected budget overrun based on burn rate | "Budget Marketing Q2 al 78% con 2 mesi rimanenti" |

### DB Schema: AiInsight

```prisma
model AiInsight {
  id            String          @id @default(cuid())
  type          InsightType
  severity      InsightSeverity // LOW, MEDIUM, HIGH, CRITICAL
  title         String
  description   String
  action_label  String?         // "Vedi dettagli", "Consolida ordini"
  action_url    String?         // Deep link to relevant page
  metadata      Json?           // Structured data for rendering
  dismissed     Boolean         @default(false)
  expires_at    DateTime        // TTL: configurable per type (see defaults below)
  created_at    DateTime        @default(now())

  @@index([dismissed, expires_at])
  @@index([type, severity])
  @@map("ai_insights")
}

// Default TTL per insight type (configurable in constants):
// SPEND_ANOMALY: 72h, VENDOR_RISK: 72h, SAVINGS: 168h (1 week),
// BOTTLENECK: 48h, BUDGET_ALERT: 336h (2 weeks)

enum InsightType {
  SPEND_ANOMALY
  VENDOR_RISK
  SAVINGS
  BOTTLENECK
  BUDGET_ALERT
}

enum InsightSeverity {
  LOW
  MEDIUM
  HIGH
  CRITICAL
}
```

### insight.service.ts

**`generateInsights()`**:
1. Query DB: spend last 90 days grouped by vendor (scoped to `tenant_id: 'default'` for Invoice queries)
2. Query DB: delivery performance last 30 days
3. Query DB: approval times by department
4. Query DB: active budget burn rates
5. Query DB: recurring items cross-vendor
6. Pass aggregated data to Claude with system prompt for analysis
7. Claude returns structured JSON array of insight cards
8. Score, deduplicate against active insights, save to DB
9. Cleanup: delete expired insights

**Error handling for Claude API failures:**
- If Claude times out or returns malformed JSON → log error, skip this generation cycle, return `{ generated: 0, error: 'claude_unavailable' }`
- If Claude exceeds token budget → retry once with reduced context (last 30 days instead of 90)
- Partial results: if Claude returns some valid insights + some invalid → save valid ones, log invalid

**`getActiveInsights()`**:
- `WHERE dismissed = false AND expires_at > now()`
- `ORDER BY severity DESC, created_at DESC`
- `LIMIT 6` (max cards in dashboard widget)

### Dashboard Widget

- Shows up to 6 insight cards in the main dashboard
- Each card: colored left border by severity, title, description, action button
- Dismiss button (X) marks as dismissed
- Action button links to relevant page

### API Endpoints

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| POST | `/api/ai/insights` | Webhook HMAC | n8n trigger: generate new insights |
| GET | `/api/ai/insights` | Session | Fetch active insights for dashboard |
| PATCH | `/api/ai/insights/:id` | Session | Dismiss an insight |

## Feature 2: AI Agent with Actions

### Interaction Flow

```
User prompt → agent.service.ts
  → Resolve user role + build permitted tool set
  → Claude with tools + conversation history (SSE stream)
  → IF READ tool → execute silently, pass result to Claude, continue stream
  → IF WRITE tool → STOP stream, emit SSE event:
      { type: "action_request", actionId, tool, params, preview }
    → Frontend shows ActionConfirmationDialog
    → User clicks Confirm → POST /api/chat/confirm { actionId }
    → User clicks Cancel → POST /api/chat/confirm { actionId, cancelled: true }
  → Server resumes: execute tool or return cancellation, continue stream
```

### SSE Protocol Extension

New SSE event types added to the existing `ChatStreamEvent` union:

```typescript
| { type: 'action_request'; actionId: string; tool: string; params: Record<string, unknown>; preview: ActionPreview }
| { type: 'action_confirmed'; actionId: string; result: unknown }
| { type: 'action_cancelled'; actionId: string }
```

### Pending Action Store

A short-lived in-memory store (Map with 5-minute TTL) holds pending actions server-side:

```typescript
// lib/ai/pending-actions.ts
// Map<actionId, { tool, params, userId, expiresAt }>
// - Action ID is a random cuid generated server-side
// - On confirm: validate actionId belongs to userId, check not expired
// - Auto-cleanup: expired entries pruned on each access
// - NOT persisted to DB (short-lived, lost on restart = safe)
```

### Confirmation Endpoint

```
POST /api/chat/confirm
Body: { actionId: string, cancelled?: boolean }
Auth: Session (must match userId who initiated the action)
```

### WRITE Tools & RBAC Matrix

| Tool | Description | REQUESTER | MANAGER | ADMIN |
|------|-------------|:---------:|:-------:|:-----:|
| `create_request` | Create new purchase request | ✅ | ✅ | ✅ |
| `update_request` | Update own request (title, description, priority, needed_by, vendor_id, category — no status transitions) | ✅ own | ✅ | ✅ |
| `submit_for_approval` | Submit request for approval | ✅ own | ✅ | ✅ |
| `approve_request` | Approve a pending request | ❌ | ✅ | ✅ |
| `create_vendor` | Create new vendor | ❌ | ✅ | ✅ |
| `bulk_update` | Bulk update requests (max 50/batch, fields: priority, category, department only) | ❌ | ❌ | ✅ |

All 8 READ tools are available to all authenticated roles including VIEWER.

### agent.service.ts

**`processAgentMessage(userId, message, conversationHistory)`**:
1. Resolve user role and permissions from session
2. Build tool set = all READ tools + WRITE tools filtered by user role
3. Call Claude with tools + conversation history
4. Handle tool_use responses:
   - READ tool → execute silently, feed result back to Claude
   - WRITE tool → return `{ status: "pending_confirmation", action, params, preview }` to frontend
5. On confirmation → execute tool handler, create audit log entry
6. On cancel → return "Azione annullata" message
7. Continue conversation with tool execution result

### ActionConfirmationDialog

- Modal dialog triggered when agent wants to execute a WRITE tool
- Shows: action name, structured preview of what will be created/modified
- Two buttons: "Annulla" (cancel) and "Conferma" (confirm with indigo accent)
- Preview renders contextually based on tool type (request fields, vendor fields, etc.)

### Audit Trail

Every confirmed agent action creates a `TimelineEvent`:
```
type: "ai_agent_action"
title: "Richiesta creata dall'AI Agent"
metadata: { tool, params, user_confirmed: true, agent_reasoning }
actor: userId
```

## Feature 3: Inventory Forecasting

### Two Prediction Modes

**Default — Weighted Moving Average (free, instant)**:
- Consumption data: last 6 months per material
- Weights (configurable in `lib/constants/forecast.ts`): month -1 = 3, -2 = 2.5, -3 = 2, -4 = 1.5, -5 = 1, -6 = 0.5
- Output: projected consumption for next 3 months, days remaining at current rate
- Cost: €0 | Latency: ~50ms
- Good for: stable, predictable consumption patterns

**On-Demand — Claude AI (precise, paid)**:
- Everything above + additional context:
  - Seasonality (year-over-year comparison)
  - Open projects/requests (upcoming demand)
  - Vendor trends (lead time, reliability)
- Claude structured output: prediction, confidence score, reasoning, risks
- Cost: ~€0.02/analysis | Latency: ~3s
- Good for: irregular patterns, strategic planning

### forecast.service.ts

**`getBasicForecast(materialId)`**:
- Query: 6-month consumption from RequestItem joined with Material
- Calculate WMA locally
- Return: `{ projected: number[], daysRemaining, reorderNeeded }`

**`getAiForecast(materialId)`**:
- Everything in basic + context queries (projects, seasonality, vendor performance)
- Claude structured output
- Return: `{ ...basic, aiProjected, confidence, reasoning, risks }`

**`checkReorderAlerts()`** (called by n8n cron daily at 8:00):
- For each material with `min_stock_level` defined (existing field in Material model):
- Compute current stock: `SUM(StockLot.current_quantity) WHERE status = 'AVAILABLE' AND material_id = X`
- If computed stock <= `min_stock_level` → create MaterialAlert
- Deduplicate: don't create alert if active (non-dismissed) one exists for same material

**Note on existing schema mapping:**
- `reorder_point` → uses existing `Material.min_stock_level` (Decimal, already in schema)
- `stock_current` → computed at query time from `StockLot.current_quantity WHERE status = AVAILABLE`
- No schema changes needed on Material or StockLot models

### Auto-Reorder: Suggestion Banner

When stock drops below reorder point, a banner appears on the Materials page:
- Shows: material name, days remaining, suggested quantity, suggested vendor (last used)
- "Crea richiesta →" button opens request form pre-filled with material, quantity, vendor
- "Ignora" button dismisses the alert
- Alert auto-resolves when a new request is created for that material

### DB Schema: MaterialAlert

```prisma
model MaterialAlert {
  id                  String    @id @default(cuid())
  material_id         String
  material            Material  @relation(fields: [material_id], references: [id])
  type                AlertType // REORDER_SUGGESTED, LOW_STOCK, OUT_OF_STOCK
  suggested_qty       Int?
  suggested_vendor_id String?
  suggested_vendor    Vendor?   @relation(fields: [suggested_vendor_id], references: [id])
  days_remaining      Float?
  dismissed           Boolean   @default(false)
  resolved_by         String?   // request_id that resolved the alert
  created_at          DateTime  @default(now())

  @@index([material_id, dismissed])
  @@map("material_alerts")
}

enum AlertType {
  REORDER_SUGGESTED
  LOW_STOCK
  OUT_OF_STOCK
}
```

### API Endpoints

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| GET | `/api/ai/forecast/:materialId` | Session | Basic WMA forecast |
| POST | `/api/ai/forecast/:materialId` | Session (rate limited: 10/user/hour) | AI forecast on-demand |
| GET | `/api/ai/forecast/alerts` | Session | List active alerts |
| POST | `/api/ai/forecast/check` | Webhook HMAC | n8n cron: recheck all materials |
| PATCH | `/api/ai/forecast/alerts/:id` | Session | Dismiss or resolve alert |

## Dependencies & Constraints

- **Anthropic SDK** already in `package.json` (used by chat.service.ts)
- **No new external dependencies** required
- **Prisma schema changes**: 2 new models (AiInsight, MaterialAlert), 3 new enums, relations added to Material and Vendor
- **n8n**: 2 new workflows (cron-based, no complex logic)
- **Cost estimate**: ~€5-15/month for moderate usage (insights every 6h + occasional AI forecasts)
- **Tenant scoping**: Currently single-tenant (`tenant_id: 'default'`). Invoice queries in insight.service.ts must filter by `tenant_id: 'default'`. New models (AiInsight, MaterialAlert) do NOT have tenant_id since they are derived data, not source data. If multi-tenancy is added later, these models will need a migration.

## Testing Strategy

- **Unit tests**: Each service function with mocked Prisma + Claude responses
- **Integration tests**: API endpoints with seeded test data
- **RBAC tests**: Verify tool filtering per role in agent.service.ts
- **Forecast accuracy**: Validate WMA calculation against known datasets
